# v0.5.25

> 这一版只补一件事：修一个 Read 工具会让整条会话"卡 400"循环的真 bug——文件扩展名是 `.png` 但内容不是真 PNG 时（常见路径：`curl -s` 写错 URL 拿到的 404 JSON 被存成 `.png`），Read 之前会把这段非图字节包成 image block 发给模型 API，Bedrock 返回 `IMAGE_FORMAT_UNSUPPORTED`，坏块留在历史里，每一轮重发都会再撞同一个 400，会话就锁死了。

## 修复

- **FileReadTool 加 magic byte 校验，拒绝伪装成图片的非图文件**（#907）。`readImageWithTokenBudget` 之前只看扩展名就把字节打包成 `image` content block，遇到 "扩展名是 `.png` 但内容不是 PNG" 的文件就会让 Bedrock 抛 `IMAGE_FORMAT_UNSUPPORTED`；坏 block 留在 conversation history 之后，模型每一轮重发上下文都会再次触发同一个 400，会话进入循环报错状态停不下来。本版在空文件检查之后、宽松的 `detectImageFormatFromBuffer` 之前加一层严格的 PNG/JPEG/GIF/WebP magic-byte 校验，magic 不匹配时抛新增的 `InvalidImageDataError`，错误信息包含文件路径、实际字节数、前 16 字节 hex 以及处理建议（重新拿源数据或当文本读），让模型下一轮能自己 recover、不再循环。

## 验证

- 新增 `src/tools/FileReadTool/FileReadTool.imageValidation.test.ts`，4 个用例：
  - 真实 bug 重现：43 字节 JSON 错误体写成 `.png` → 抛 `InvalidImageDataError`，message 含路径 / size / `7b226f6b`（`{"ok` 的 hex）。
  - 3 字节截断 buffer → 抛错。
  - 空文件 → 保留既有的 `Image file is empty` 错误（回归保护）。
  - 1×1 真 PNG → 正常通过。
- `bun test src/tools/FileReadTool/ src/tools/AgentTool/` → 169/169 全绿，邻居 scope 零回归。
- 独立 verification subagent 跑了 12 个对抗探针（PNG-off-by-one、RIFF-WAVE-not-WEBP、BMP、HTML/JSON 伪装 `.png`、截断、真 JPEG-JFIF、GIF87a/GIF89a 等）全部分类正确。

## 升级注意

- 行为变化只在"读到扩展名是图但内容不是图"的文件时可见——以前会让模型 API 在下一轮 400，现在 Read 调用本身以清晰错误结束（错误信息带定位线索），模型能自己换策略。
- `detectImageFormatFromBuffer` 的语义未改（仍然在 unknown 输入时默认 `image/png`），它的其它三个调用方不受影响。

## 范围

- `src/tools/FileReadTool/FileReadTool.imageValidation.test.ts`：新增回归测试。
